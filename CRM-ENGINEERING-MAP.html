CRM Portal Onboarding Checklist - Engineering Map

Precise technical reference for building a shop-facing onboarding checklist view

1. PORTAL AUTHENTICATION

JWT Token Generation & Verification

/lib/portal-token.ts
Secret: PORTAL_JWT_SECRET env var (required)
Expiry: 30 days
Type field: 'portal' for capabilities portal

PortalJwtPayload structure:

type PortalJwtPayload = { locationId: string type?: string // 'portal' for new tokens }

Token signing (line 12-13):

jwt.sign({ locationId, type: 'portal' }, secret(), { expiresIn: '30d' })

Verification function (line 16-22):

verifyCapabilitiesPortalToken(token: string): { locationId: string } - Validates token type === 'portal' - Returns { locationId } - Throws on invalid/expired tokens

Token Generation Route

/app/api/portal/generate-token/route.ts

POST /api/portal/generate-token

  • Requires NextAuth session (internal: BDR/admin only)
  • Request body: { locationId: string }
  • Calls signCapabilitiesPortalToken(locationId)
  • Response: { token: string, portalUrl: string }

Portal Data Fetching

/app/api/portal/location/route.ts

GET /api/portal/location?token=...

  • Token verified via verifyCapabilitiesPortalToken(token)
  • Uses supabaseAdmin (service role)
  • No RLS checks - relies on JWT + service role
  • Selects 27 location fields including capabilities
  • Returns location + primary contact data

Autosave (Patch Updates)

/app/api/portal/autosave/route.ts

PATCH /api/portal/autosave

  • Token verified via verifyCapabilitiesPortalToken(token)
  • Uses supabaseAdmin (service role)
  • Validates each field in patch object
  • Updates locations table

Security Summary

Auth Mechanism: JWT (stateless)
Supabase Client: Service role key (admin client)
RLS: NOT used - JWT is the boundary
Mapping: One JWT → one location_id (1:1)
Scope: Can read/write location record for that location_id only

3. CHECKLIST APIs (READ & UPDATE)

Request body:

{ item_key: string // required, lowercase completed?: boolean // true = mark done notes?: string | null // optional notes }

Auth: NextAuth session (internal admin only)

Checklist Upsert Logic (lines 71-83):

const checklistPatch = { enrollment_id: id, item_key: itemKey, completed_at: completed ? new Date().toISOString() : null, completed_by_user_id: completed ? session.user?.email : null, updated_at: new Date().toISOString() } supabaseAdmin .from('program_enrollment_checklist') .upsert(checklistPatch, { onConflict: 'enrollment_id,item_key' })

Authorization

Current: All checklist PATCH requires NextAuth session (internal admin/FL only)
No portal user can call checklist endpoints today. This is a new capability for onboarding view.

6. COMPLETION ATTRIBUTION

Current Capture

/supabase/migrations/023_tesla_program_enrollments.sql (line 34)

Field: completed_by_user_id (text)

How it's set (from /app/api/tesla/enrollments/[id]/checklist/route.ts, line 67):

const completedBy = completed ? (session.user?.email ?? null) : null checklistPatch.completed_by_user_id = completedBy

Stores the email of the NextAuth user who made the PATCH request.

Limitations

No distinction of WHO completed it:
  • No user ID (only email string)
  • No role/type field (internal vs shop user)
  • No "completed by" name or title

For shop-facing onboarding view, extend schema with:

  • completed_by_source: 'admin' | 'portal' | 'system'
  • User metadata (name, role, type)
